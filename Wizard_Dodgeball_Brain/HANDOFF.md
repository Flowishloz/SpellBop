## 🐞 BUG GRAVEYARD (RESOLVED — DO NOT REINTRODUCE)
- **Frame-0 backwards fireball (Sprint 2 hotfix, 2026-06-10):** Godot editor fails to persist SG Physics 2D collision layer/mask UI assignments — all bodies silently fell back to layer 1/mask 1, so spawned fireballs collided with their caster and bounce()d backwards on tick 0. FIX: layers/masks hardcoded in `_ready()` via `scripts/physics_layers.gd` constants (Players=1, Projectiles=2, Walls=4). RULE: never assign SG collision layers through the Inspector. Regression test: `tests/test_collision_layers.gd`.
- **Dead spawn_offset_y export (Sprint 2 hotfix, 2026-06-10):** float @exports cached to fixed-point once in `_ready()` ignore runtime Inspector edits — symptom: "changing the export does nothing". FIX: `spawn_offset_y` now uses a setter that re-caches on write. RULE: any float @export whose fixed-point cache should respond to live tuning needs a re-caching setter. Also note: `scenes/test_area.tscn` overrides `spawn_offset_y=24.0` on its Player instance, shadowing `player.tscn` (10.0) and the script default (48.0) — per-instance scene overrides shadow tuning edits made elsewhere. Regression test: `tests/test_spawn_offset.gd`.
- **Origin spawn trap — local vs global fixed coordinates (Sprint 2 hotfix 2, 2026-06-10):** SpellCasterComponent read the caster's LOCAL `fixed_position` and wrote it into the projectile container's coordinate space — with the player positioned via a parent/wrapper or a container at a different origin, fireballs spawned at midfield (0,0). FIX: spawn path is global end-to-end via `get_global_fixed_position()` / `set_global_fixed_position()` (SG API has getter/setter METHODS, no property). RULE: never mix SG local fixed_position across different parents — cross-node placement must use the global fixed API. Regression test: `tests/test_spawn_offset.gd` TEST 4.
- **Frame-0 facing default (+Y) (Sprint 2 hotfix 2, 2026-06-10):** `cast_direction_y` defaulted to +1, but down-court toward the opposing wall is sim −Y — an unconfigured caster fired at its own baseline on frame 0. FIX: default is now −1 (player wakes facing North/down-court); P2 overrides to +1. RULE: directional defaults must point down-court (−Y).
- **"Stuck at Y=0" — editor float transform never reaches the sim (Sprint 2 hotfix 3, 2026-06-11):** SG Physics 2D keeps the float Node2D transform and the fixed-point transform SEPARATE — dragging a node with the editor gizmo or setting float position/rotation writes ONLY the float side; the deterministic sim reads ONLY fixed_position/fixed_rotation. The player "positioned at 12 m" via the gizmo was at fixed (0,0) the whole time. No origin-teleport code ever existed (three audits + headless probe). FIX: none needed in code. RULE: position/rotate SG nodes ONLY via the fixed_position_x/y / fixed_rotation Inspector fields (raw 64.16: sim units × 65536) — never the 2D viewport gizmo.
- **Physics rotation lock (Sprint 2 hotfix 3, 2026-06-11):** mandated guard — PlayerController._ready() hardcodes fixed_rotation = 0. Character facing is visual-only on the 3D rig. RULE: never rotate/mirror/negative-scale the SGCharacterBody2D or any SG ancestor for facing. Regression test: tests/test_spawn_offset.gd ROTATION INVARIANT checks (cast direction proven invariant to a 180° body rotation and to movement history).
- **Facing-yaw orbit teleport (Sprint 2 hotfix 3, 2026-06-11):** the new rig facing flip (visual_root.rotation.y = 0/PI on A/D) visually teleported the character across the arena because the rig content was not centered on the yawed node's origin — a 180° yaw ORBITS offset content, it does not flip it in place. FIX: face_movement_direction now defaults to FALSE and the yaw is applied to a dedicated @export facing_pivot (falls back to visual_root). RULE: before enabling facing, center the WizardRig on facing_pivot's origin and verify by yawing 180° in the editor — the wizard must spin in place, not move.
- **Sim/visual split-brain — sprites placed instead of bodies (Sprint 2 hotfix 4, 2026-06-11):** the wizard Sprite3D was hand-positioned at z=+10.9m INSIDE the rig while the sim body sat at fixed y=+12 sim units (0.12m, arena center); the wall visual drifted to z=-55m while its collider sat at y=-60 units near center. Result: fireballs spawned (correctly!) at the BODY — which was at the arena center, ~11m behind the art — and instantly hit the wall collider 3 ticks away ("instant bounce-back from center"), while side casts missed the 600-unit-wide collider entirely. FIX: sprites zeroed inside their rigs; sim bodies repositioned to where the art stands (player fixed y=+1100 units -> rig z=+11m; wall fixed y=-1000 -> z=-10m); colliders resized to the measured art (fireball radius 50 units = 1m orb; wall 1152x60 units = 11.5m sprite at pixel_size 0.003; player footprint 40x24 units); arena_half_width 500 to match the 10m floor. RULE: the Node3D rig must sit at the BODY's mapped position (sim x 0.01) with art centered on the rig origin — never hand-place sprites to "fix" a misplaced body.
- **One-frame center flash on projectile spawn (Sprint 2 hotfix 4, 2026-06-11):** a spawned projectile's visual rendered one frame at the scene-default origin because the first state_updated only fired on the next physics tick. FIX: ProjectileMovementComponent.launch() now emits state_updated immediately with the body's spawn position. RULE: any sim entity teleported at spawn must emit its position signal before the next render.
- **Invisible 3D world — .tscn Transform3D is ROW-major (Sprint 3 hotfix, 2026-06-11):** the hand-written match_arena camera basis was authored as basis COLUMN vectors, but .tscn `Transform3D(...)` serializes basis ROWS — the transposed matrix pitched the camera 33 degrees UP at the empty sky, rendering only the default clear color. Proven by an automated screenshot probe (tests/screenshot_probe.gd renders the scene and saves a PNG). FIX: rebuilt camera with an IDENTITY basis (Punch-Out perspective, no rotation). RULE: never hand-author rotated Transform3D values in .tscn — use identity bases positioned by origin, or set rotation from script; ALWAYS verify scene composition with the screenshot probe before handing off.
- **Freed-instance == null re-capture (Sprint 3 tuning, 2026-06-11):** in Godot 4 a FREED object compares equal to null, so a test that guarded "capture once" with `if ref == null:` silently re-captured a fresh instance every time the old one was freed — masking a working lifespan feature as a test failure. FIX: capture-once guards use a dedicated bool flag, never a null check. RULE: never use `== null` to mean "not yet assigned" on refs that may be freed; pair is_instance_valid() with an explicit captured flag.
- **Idle-caster speed-scale erasure (Card sprint, 2026-06-11):** with TWO casters pushing movement penalties through a single-writer set_speed_scale, the idle caster's "no penalty = ONE" write erased the active caster's penalty every tick (last writer wins in child order). FIX: MovementComponent.apply_speed_penalty() MIN-composes into an accumulator that self-resets each tick — active casters re-push every tick, idle casters push nothing. RULE: shared per-tick modifiers must compose (min/sum), never overwrite.
- **Shared sub-resource mutation (Card sprint, 2026-06-11):** instanced scenes SHARE their sub-resources — mutating a spawned barrier/fireball's collision shape (extents/radius) would silently resize EVERY live instance of that scene. FIX: BarrierController.deploy() and FireballController.apply_size() duplicate() the shape before writing. RULE: never mutate a scene sub-resource on an instance without duplicating it first.
- **Vector2 value-type spring helper (Sprint 6, 2026-06-12):** a shared spring function taking (pos, vel) Vector2 parameters silently mutated COPIES — Vector2/float are value types in GDScript parameters. Symptom: UI springs do nothing. FIX: springs are inlined against member variables at every call site. RULE: never write helpers that "mutate" value-type parameters — return values or operate on members.
- **Anchor presets before parenting (Sprint 6, 2026-06-12):** calling set_anchors_preset() on a Control BEFORE add_child(), then setting position/size, recomputed offsets on tree entry and shoved a full-width banner off-center (caught by the screenshot probe). FIX: UI built in code uses explicit position/size with default (0) anchors — canvas space is a stable 1080x1920 under the canvas_items stretch. RULE: in code-built UI, skip anchor presets or apply them only after the node is in the tree.
- **Sim-tick stage countdown vs wall-clock window desync (Sprint 7, 2026-06-12):** the staged spell's release was counted in SIM ticks while the on-screen countdown ran on the wall-clock Stack window — two clocks for one event drifted (playtest: "the spell releases as soon as the card appears"). FIX: the window closing IS the release — MatchController resolves the stack LIFO from TheStack.stack_closed; the sim-tick countdown was deleted. RULE: one event, one clock — never pace the same gameplay beat with both sim ticks and wall time.
- **Container tick-order capture miss (Sprint 11, 2026-06-12):** sibling entities in the Projectiles container tick in CHILD ORDER, so whether the barrier scans before or after a ball moves depends on spawn order — a fast ball could bounce and travel a full tick-step out of the barrier's capture band between scans ("the WOA anticipation sometimes doesn't fire"). FIX: proximity bands are SPEED-AWARE (widened by the target's per-tick velocity). RULE: any per-tick proximity scan against entities that move a comparable step per tick must widen its window by that step — never assume scan-before-move ordering between container siblings.
- **Slow-mo tap loss (Sprint 12, 2026-06-12):** inside the 10% Stack window the sim samples input at ~6 Hz — a quick Shift/card tap fell entirely between two ticks and vanished ("the dash waits for the slow-mo to finish"). FIX: PlayerController polls press edges at RENDER rate and latches them into the next tick's input dict. RULE: edge-triggered actions must be collected at render rate and latched for the sim — never sampled only at tick rate.
- **Bare autoload identifiers in standalone compilation (Sprint 12, 2026-06-12):** scripts referencing the SoundFX autoload by name compiled in-game but FAILED --check-only/standalone loads ("Identifier not found") — the same console-noise class as the earlier editor spam. FIX: a class_name static gateway (Sfx.play) resolves the autoload defensively at call time. RULE: cross-cutting autoloads referenced from many scripts get a class_name static gateway; direct autoload identifiers only inside scenes that guarantee the autoload exists.
- **Freed player in a typed assignment — SFX crossfade crash (Sprint 13 hotfix, 2026-06-12):** SoundFX cached the last AudioStreamPlayer per sound; a naturally FINISHED player frees itself while the cache still holds the dead reference, and pulling that into a TYPED variable (`var prev: AudioStreamPlayer = dict.get(...)`) is a hard script error ("Trying to assign invalid previously freed instance") — crashing on the next retrigger of the same sound. Headless suites never caught it because the dummy audio driver never finishes a stream. FIX: retrieve UNTYPED (Variant), gate on is_instance_valid, only then cast; the finished callback also erases its own cache slot. Repro: tests/manual_sfx_crash_repro.gd. RULE (extends the Sprint 3 freed-instance rule): caches of freeable nodes must be read into UNTYPED variables and validity-checked BEFORE any typed assignment or cast — a typed declaration alone is a crash, not a null.
- **First-draw pipeline compile hitch (Sprint 14, 2026-06-12):** scenes/materials instantiated mid-fight (the stack resolution spawning projectiles, BurstFX building a fresh StandardMaterial3D per burst) compile shaders/render pipelines on FIRST DRAW — a visible lag spike exactly at the slow-mo exit. FIX: MatchController warms every projectile/barrier scene + burst type with an occluded draw at round start; BurstFX caches meshes/materials per look and duplicates compiled templates. RULE: anything that can be instantiated mid-fight must be drawn once during a loading beat; never construct fresh materials inside gameplay-time spawns.
- **Label autowrap property-order trap (Sprint 16, 2026-06-12):** setting a Label's `size` BEFORE enabling `autowrap_mode` snaps the control to its single-line minimum width, so the wrap never engages — descriptions rendered as one clipped line (found only by cropping/upscaling a screenshot). Related: `clip_text` is a single-line ellipsis mode, not rect clipping — wrapped labels need `clip_contents`. RULE: for code-built wrapped labels, set autowrap/clipping/font FIRST, size second, text last; and verify text layout in PIXELS, not in code review. Also logged: a stored-and-killed pattern for transient UI tweens (a stale ROUND-N fade ate the VICTORY title).
- **No Android binary for the native sim extension (Sprint 17, 2026-06-13):** sg-physics-2d — the GDExtension the ENTIRE deterministic sim depends on (SGFixed, SGCharacterBody2D, etc.) — ships ONLY windows/linux x86_64 binaries upstream; there is NO Android build for ANY release version (confirmed via the gitlab releases API). A naive Android export installs fine but HARD-CRASHES the instant a match scene loads, because the arm64 .so is missing. FIX: cross-compiled sg-physics-2d v1.0.0-alpha13 (the exact version the project uses) from gitlab source for android arm64 — Android NDK r25c, the godot-cpp submodule, scons platform=android arch=arm64 for BOTH template_debug and template_release; dropped the two .so files into addons/sg-physics-2d/lib/ and added android.debug.arm64 / android.release.arm64 entries to sg-physics-2d.gdextension. RULE: before exporting to ANY new platform, every GDExtension the project depends on must have a native binary for that platform's arch/target — verify by unzip-listing the exported APK/IPA and confirming the .so is under lib/<abi>/. Same will apply to iOS later (needs an ios arm64 build of sg-physics-2d).
- **delta_rollback was dead weight, not a dependency (Sprint 17, 2026-06-13):** the delta_rollback GDExtension was loaded at startup but its classes (SyncManager etc.) are referenced ONLY in code COMMENTS as a future plan — never instantiated; the game runs on PlayerController's own local tick driver. Its .gdextension even declared android libs whose .so files do not exist on disk, so it would have errored on device. FIX: disabled it by renaming addons/delta_rollback/bin/delta_rollback.gdextension -> .gdextension.disabled (and its .uid). All 6 original suites still pass and determinism stays BIT-IDENTICAL, proving nothing depended on it. RULE: a GDExtension sitting in addons/ loads even if its "plugin" is not enabled; disable unused ones before mobile export so they don't demand per-platform binaries.
- **Projectiles/walls spawned outside the arena (Sprint 17 hotfix, 2026-06-13):** every spawn site (_resolve_attack/_resolve_counter/_resolve_defense in card_caster_component.gd, _spawn_projectile in spell_caster_component.gd, split_on_barrier in fireball_controller.gd) placed the object at caster_pos.x with NO bounds clamp, so a WIDE object (Icey Retort frost wave, ±200u) spawned while the caster sat at the lane edge reached ~150u past the side walls. The aim mechanic was a red herring: _aim_vx_fp tilts launch VELOCITY only, never spawn position — but holding a direction to bank aim parks the caster at the edge, so it correlated. FIX: a deterministic fixed-point spawn-X clamp — MovementComponent.clamp_spawn_x_fp(x, half_extent, bound) (STATIC, pure clampi on 64.16 ints, limit floored at 0) + MovementComponent.arena_half_width_fp() — applied at all five sites, clamping the object's FULL extent (centre ± half-extent) inside the caster's MovementComponent bound (read at spawn time so scene overrides win); aim/velocity untouched so spells stay aimable. Regression: tests/test_spawn_bounds.gd. RULES: (a) clamp every spawned gameplay object's spawn ORIGIN so its full collider extent stays inside the arena bound, using the POST-resize half-extent (apply_size/deploy) and the scene's MovementComponent bound — never a hardcoded constant; (b) aim is VELOCITY ONLY and must NEVER become a spawn-position factor (reinforces the spawn-offset graveyard rule). FORWARD-LOOKING CAVEAT (from the adversarial review): the bound is currently a SHARED global property (both wizards = 400, symmetric court), so the shard reading "the first wizard's mover" and the casters' from_float(400) defensive fallback are correct today. If arena bounds ever become PER-PLAYER asymmetric, revisit split_on_barrier's bound lookup (have the shard carry its caster's bound) and the fallback constants.
- **Wizard sprite sank into the floor — animator zeroed the sprite offset (Sprint 18 follow-up, 2026-06-14):** the charge-shake in wizard_animator_component.gd wrote `_sprite.position = Vector3(jitter)` while charging and reset it to `Vector3.ZERO` otherwise, EVERY frame — overwriting the Sprite3D's authored local offset (0, 1.13, 0) and dropping the sprite centre to y=0, so its lower half rendered below the floor plane. FIX: capture `_sprite_base_pos = _sprite.position` in _ready() and offset jitter from that base (reset TO the base, never to zero). Compounding cause: the sprite used full billboard (mode 1), which tilts ~23deg toward the down-angled camera and dips the bottom into the floor — switched to Y-axis billboard (mode 2, stays upright). RULE: a transient visual effect (shake/jitter/recoil) must offset from a CAPTURED base transform, never assign an absolute value or reset to zero — that silently destroys authored offsets. For billboard characters on a floor plane, use Y-billboard so they stay upright and their base stays on the ground.
- **Two headless Godot instances deadlock on the import cache (Sprint 19, 2026-06-14):** running two `godot --headless` processes against the SAME project at once HANGS both (0 output, never exits) — they contend on the `.godot/` import cache/lock. Symptom: a test that ran fine alone "hangs" when another instance is live. FIX/RULE: run the headless test suites STRICTLY SEQUENTIALLY (one finishes before the next starts); never fan them out in parallel.
- **Stale pickup reference after a spawner refresh (Sprint 19, 2026-06-14):** the healing-emerald spawner REFRESHES (frees the un-claimed emerald and spawns a fresh one) every interval, so a test holding a captured emerald node reference was poking a FREED instance after the refresh (with a fast test cadence). FIX/RULE: in tests, freeze the spawner after capture (set MatchController.emerald_scene = null) or re-fetch the live emerald from the "pickups" group each time — never hold a long-lived reference to a refreshable spawned entity.
