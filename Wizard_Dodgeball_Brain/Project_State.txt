[SPRINT 17 HOTFIX — SPAWN BOUNDS: NO PROJECTILE SPAWNS OUTSIDE THE MAP (2026-06-13)]
Bug (Creative Director): "Icey Retort's counter wave sometimes spawns outside the map — possibly the aiming mechanic based on last input. Keep it aimable, but NO projectile should spawn outside the map."
Root cause: every spawn site placed the object at caster_pos.x with NO arena-bounds clamp. The aim mechanic (_aim_vx_fp) only tilts launch VELOCITY, never spawn position — so the out-of-bounds was the WIDE object's centre sitting at the lane edge (where holding a direction to bank aim also parks the caster). The Icey Retort frost wave is ±200u half-width (ice_wave.tscn); spawned at the +400u player edge its body reached ~+600u, ~150u past the ±450u side walls.
Fix (deterministic, rollback-safe, fixed-point): added MovementComponent.arena_half_width_fp() accessor + a STATIC MovementComponent.clamp_spawn_x_fp(x_fp, half_extent_fp, bound_fp) that returns clampi(x, -(bound-half), +(bound-half)) with the limit floored at 0 (objects wider than the lane pin to centre). Applied at ALL FIVE spawn sites, clamping the spawn ORIGIN only so the object's FULL extent stays inside the arena bound; launch velocity (the aim tilt) is left untouched so spells stay fully aimable:
  1. card_caster_component.gd _resolve_counter (Icey Retort wave, half = card.projectile_size 200) — THE HEADLINE FIX.
  2. card_caster_component.gd _resolve_attack (Spark Bolt fan, half = card.projectile_size 31; clamped once before the fan loop since the fan is velocity-only).
  3. card_caster_component.gd _resolve_defense (Gaea's Wall, half = card.wall_size.x*0.5 = 102).
  4. spell_caster_component.gd _spawn_projectile (base fireball, half = spell.projectile_size 24).
  5. fireball_controller.gd split_on_barrier (barrier-breaker shards, half = BASE_RADIUS_UNITS*0.75 = 18, clamped per shard AFTER the ±28 fan offset).
Bound source: each caster reads its sibling MovementComponent.arena_half_width_fp() AT SPAWN TIME (post-_ready, always resolved) so scene overrides win (400 in match_arena via player.tscn; 500 test_area; 600 script default); the projectile shard (no mover of its own) reads the bound from the "wizards" group deterministically (scene-tree order). The clamp targets the per-caster MOVEMENT bound (±400), which is 50u inside the physical ±450 walls — strictly inside the map.

[2026-06-15] ROUND-2 NETPLAY DESYNC FIXED — WARM-UP BARRIER IN THE SIM PHYSICS (commit d3a53a4)
-----------------------------------------------------------
The round-2 FREEZE from the CD A/B/B2 APK playtest (the entry above) is FIXED + fully verified on branch
feature/spawn-rollback-cards. It was a rollback DESYNC — NOT cross-platform determinism (same build) and NOT a
crash (no script error). With this fixed, item C is UNBLOCKED and is the SOLE remaining mobile-MP task.

ROOT CAUSE (the surprise — NOT the aim/spawn/bounce sim, which were ALL deterministic): the render-pipeline
WARM-UP.
- MatchController._warm_up_render_pipelines() (the Sprint-14 first-draw-hitch fix) add_child()s barrier.tscn +
  the projectile scenes to warm shaders, moving only their VISUAL Node3D under the floor. But each scene ROOT
  is an SG physics body, so its _ready() REGISTERS it in the deterministic SG physics space — and the unmoved
  barrier COLLIDER sat on collision_layer = LAYER_WALLS (4) at sim (0,0) = arena CENTRE (x in [-150,150],
  y in [-20,20]).
- Those warm-up bodies are freed on a WALL-CLOCK SceneTreeTimer(0.4); the Stack slow-mo a staged card opens
  (Engine.time_scale = 0.1) STRETCHES that 0.4 s far into the match. A gameplay fireball's mask includes WALLS,
  so an aimed shot crossing centre BOUNCED off the warm-up barrier at mid-court (vx flips, X-normal, |v|
  preserved where NO real wall exists).
- Because the barrier's lifetime is WALL-CLOCK, its presence at a given SIM TICK differed between peers under
  asymmetric rollback -> one peer reflected the ball, the other didn't -> "Fatal state mismatch" -> peer torn
  down -> ping-spam 30k+ -> both apps froze. (Matches the repro: wizards in lockstep, projectiles diverge — host
  nproj=8 at x~+427, client nproj=4 at x~-153.)
- Why it HID: the plain determinism smoke never moves/aims (straight shots at x=0) and never stages a card (no
  slow-mo -> the warm-up bodies are freed at ~t=24, before the first cast at t=40), so a gameplay ball never met
  the warm-up barrier. The bounce-VFX node-spam balloon (commit 0f24d23, nproj 6->32+) was a SEPARATE real bug
  surfaced during the hunt — bounding it (nproj 32->8) did NOT fix the desync.

THE FIX (commit d3a53a4):
- In _warm_up_render_pipelines, after add_child(inst), make every warm-up body collision-INERT —
  collision_layer = 0, collision_mask = 0, sync_to_physics_engine() (guarded by "collision_layer" in inst) — so
  it renders/warms pipelines WITHOUT ever touching the rollback sim.
- ALSO FIXED (latent SMOKE-test FALSE-FAIL, same commit): core/network_manager.gd _smoke_fingerprint() sorted
  projectiles by a.name, but Node.name is a StringName and StringName < StringName compares by the interned
  POINTER (allocation-order-dependent -> non-deterministic across two processes), so the two peers listed
  identical balls in a different ORDER -> a false FAIL even in perfect lockstep. Fixed to
  String(a.name) < String(b.name). The fingerprint was also enriched (per-ball velocity + mask + a barrier dump
  nbar; nproj now filters to real SG bodies so wall-pulse VFX no longer inflate the count).
